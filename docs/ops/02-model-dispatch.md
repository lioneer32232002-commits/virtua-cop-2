# C — 模型調度守則

> 給主對話模型（未來每個 session 的「指揮官」）。目標：用最少 token 換最高正確率。
> 寫成**模型無關的原則** + 文末**本專案實際艦隊的具體階梯**。原則若和 `01-harness-diagnosis.md` 判準衝突，以更具體者為準。
> 註：本專案 CLAUDE.md 現行政策是「一律 Opus」。本檔仍寫全階梯，因為 (a) 政策可能變、(b) 就算全用 Opus，「指揮官不下場／派工三件套／驗證不自驗」照樣省 token、增正確率。

---

## 1. 指揮官不下場（Commander stays up）

主對話的 context 很貴、會被之後每一步繼承。**凡是「讀很多、找很久、可平行、會弄髒 context」的活，派 subagent，主對話只收結論。**

**一定派下去：**
- 掃 repo／大量 grep／讀多檔找答案（用 `Explore` agent）。
- 查網頁／查外部資料（subagent + WebSearch/WebFetch）。
- 批次改檔（同模式套多處）。
- 對抗式審查、read-back 驗證（見 §6）。

**指揮官自己做：**
- 決策、綜合、排序、寫給使用者的話。
- **品味/判斷題的核心**（例：定 spec、判對味、寫這種制度檔）——這種不要整包丟給弱模型，弱模型補不了品味（見 `03-judgment-rubrics.md` 與誠實條款）。
- 小到不值得一次 round-trip 的單檔小改。

**反例（別這樣）：** 指揮官自己 `Read` 十個檔拼答案 → context 爆、後面每步變慢變貴。**正解：** 一個 `Explore`／`general-purpose` subagent 去讀，回一段結論 + `檔案:行號`。

---

## 2. 派工三件套（每次 dispatch 都要齊）

漏任何一件，subagent 就會亂猜或做過頭。

1. **目標與動機**：要什麼、為什麼要、它在整體的哪個位置（scene-setting）。動機讓它遇到你沒想到的狀況能對齊你的意圖。
2. **驗收條件**：怎樣算完成、邊界在哪、**不要做什麼**（防過度建置）。可量化就量化（測試幾個、體積上限、檔案清單）。
3. **回報格式**：要它回什麼、多長、用什麼結構（見 §5 回報合約）。

> 現成模板在 `04-delegation-templates.md`，填空即用（搜尋／實作／重構／研究／審查各一份）。

---

## 3. 顯式指定 model 與 effort（不憑印象）

派工時**明講**用哪個模型、什麼 effort。不指定＝繼承，常常過殺或不足。

- **effort 檔位**（`low`/`medium`/`high`/`xhigh`/`max`）：這是 **Workflow `agent()` 的 `effort` 參數**＋「任務難度」詞彙，**不是** Agent tool 的參數。用 Agent tool 派 subagent 時**沒有 effort 旋鈕**——這時 effort 只當你心裡的難度分級（用來決定派哪個 model、驗收寫多細）。機械活＝低；一般整合＝中；最難的審查/判斷＝高。
- **配法（原則）**：
  - 觸 1–2 檔、規格完整、無判斷 → 最便宜模型、`low`。
  - 多檔整合、要對慣例、要 debug → 中階、`medium`。
  - 架構/設計/最終審查/品味判斷 → 最強模型、`high`~`max`。

（Agent tool 的 `model` 參數可填 `haiku`/`sonnet`/`opus`；**effort 只有 Workflow 的 `agent()` 吃**——Agent tool 沒這參數，別去填一個不存在的。）

---

## 4. 升降級路徑（省 token 的核心）

**升級（往強走）：**
- **便宜模型錯一次就升**——不要讓它在同一題上第二次嘗試（便宜模型的第二次通常還是錯，純燒 token）。
- **中階模型同一子任務連錯兩次 → 升級**，並把**完整失敗軌跡**（它試了什麼、錯在哪、測試輸出）一起交給更強的模型，別讓它從零重來。
- 訊號：反覆卡同一處、回報自相矛盾、驗證一直不過（見 `03-judgment-rubrics.md` §「方向錯了該換路」）。

**降級（往便宜走）：**
- 一旦強模型**解出了模式/範本**（例：改法定案、重構樣式確立），把「照這個套到其餘 N 處」**降回便宜模型批次做**——模式已知，剩下是機械套用。

**熔斷（外層上限，蓋過上面的分級規則）：** 定義「一輪」＝**一次完整嘗試 + 驗證，不分模型**。**同一個問題累計兩輪未過就停**——不是換模型硬撞，而是判斷是不是**計畫本身錯**（升級到「問使用者/改計畫」），見 `03` 的停損判準。（上面「錯一次就升／連錯兩次升」是輪內選模型的規則；這條兩輪熔斷是最外層，優先。）

---

## 5. 回報合約（subagent 回什麼）

- **只回結論**，不回過程流水帳。發現點用 `檔案:行號` 指出去，讓指揮官能跳查而不必把整段貼回主 context。
- **長產物落檔**：報告/大段程式碼/資料寫成檔，回**路徑**，不要把整包塞回對話。
- 狀態要明確：`DONE` / `DONE_WITH_CONCERNS` / `BLOCKED` / `NEEDS_CONTEXT`（含具體卡點、試過什麼、需要什麼）。
- **subagent 的回覆只給指揮官看，不是給使用者的話**——指揮官要轉譯重點給使用者。

---

## 6. 驗證不自驗（Verification ≠ self-report）

建構者說「做好了」不算數。驗收**派 fresh-context agent**（沒有建構者的盲點與 context）：

- **檔案類**：read-back——重新讀檔確認真的落地、內容完整（不是信 diff）。
- **程式碼類**：跑測試或實跑，看**真實輸出**，不接受「應該會過」。
- **高風險判斷**（架構、刪大東西、品味）：加**第二意見**，或**多答案評審選優**（N 個獨立答案 → 一個評審挑最好/找互相矛盾）。
- 本專案已驗有效：subagent-driven + spec/quality 雙審抓到過 3 個建構者盲點（clip-path 吃 glow、wipe 競態、Esc 錯過 clue）。成本考量下，機械 task 可降**單審**，品味/高風險 task 留**雙審或多審**。

**反例：** 讓建構它的同一個 agent 自己說「我檢查過沒問題」就收工 → 共享盲點，等於沒驗。
