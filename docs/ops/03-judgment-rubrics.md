# D — 判斷力外化（給弱模型的 rubric 與 checklist）

> 把「只有強模型才擅長的判斷」寫成弱模型能照跑的判準。每條附**一個正例 + 一個反例**。
> 抽象要求（「保持高品質」）等於沒寫，這裡一律給可勾選的訊號。誠實：品味/模糊題補不了，遇到時照第 6 節「補不了怎麼辦」。

---

## 1. 何時該升級模型（往更強走）

**升級，若出現任一：**
- 便宜模型在**同一題**上已錯 1 次。
- 中階模型在**同一子任務**連錯 2 次（帶失敗軌跡升級，別讓強模型從零開始）。
- 任務需要**多個都說得通的取捨判斷**（架構選型、要不要動別人的介面）。
- 要**刪/改大東西**、或改動會影響「每 session 必載」的檔（CLAUDE.md、memory 索引）。
- 回報開始**自相矛盾**，或「讀了一堆檔還是講不清系統」。

> ✅ 正例：Haiku 批次改 10 處，第 3 處測試跑不過、它改了兩次還紅 → 立刻升 Sonnet，附上它的 diff 與測試輸出。
> ❌ 反例：Haiku 卡住，指揮官叫「同一個 Haiku 再試一次、再試一次」→ 燒 3 倍 token 還是錯。同一題便宜模型第二次通常不會變對。

---

## 2. 何時算「真的完成」（Definition of Done）

**全部勾到才算 DONE：**
- [ ] 做到了驗收條件寫的**每一項**（不是「大致上」）。
- [ ] **沒做**驗收條件外的東西（沒過度建置）。
- [ ] 有**獨立驗證**過：檔案 read-back／程式碼跑測試看真實輸出（見 `02-model-dispatch.md` §6）。不是建構者自己說 OK。
- [ ] 純邏輯有 **TDD**（先寫失敗測試→實作→綠）；本專案慣例，見既有 `game/tests/`。
- [ ] 影響玩家可見畫面的改動：知道它**要走 Electron CDP 眼驗**（preview 會凍），且已截圖或明列「待使用者對味」。
- [ ] 該 commit 的已 commit、訊息寫得夠獨立審閱者看得懂。

> ✅ 正例：「C8 holding-state 完成」＝新增 3 檔、312/312 綠、tofu guard 觸發後 `fonts:build` 重生子集、單審 APPROVED、commit `9eea39f`、並註明「手機直向待使用者對味」。
> ❌ 反例：「我加了 holding 畫面，應該可以了」——沒跑測試、沒說 tofu guard、沒獨立驗、沒 commit SHA。這是「宣稱完成」不是完成。

**鐵則：先證據，後宣稱。** 說「過了/修好了/完成了」之前，先貼出驗證指令的**實際輸出**。

---

## 3. 何時該停下來問使用者（而不是自己決定）

**停下來問，若：**
- 是**品味/對味判斷**（美術風格收斂沒、好不好玩、文案對不對味、劇情走向）——本專案分工白紙黑字：Claude 研究+產草稿+控風險，**使用者判對味**。
- 需求有**多個方向且選錯代價高**（改公開站第一印象、動核心玩法、題材風控踩線）。
- 要動**題材紅線**：主題式影射 ✅／對象式影射（指名政黨領導人）❌（見 spec §13）。
- 要**合併分支、push 到公開 repo、對外發佈**——外向且難回收的動作。

**不要為了這些停**（自己決定就好、講一聲即可）：
- 有慣例或明顯預設值的選擇（檔案放哪、命名、測試怎麼寫）。
- repo 裡查得到答案的事。

> **硬規則（別誤分類）：** 凡沾到「好不好看／好不好玩／對不對味／文案措辭／劇情走向」的，**一律歸上面『停下來問使用者』**，**不得**當成這裡的「自己決定」。拿不準是不是品味題，就當它是、去問。

> ✅ 正例：Phase C 做完，跑完自查後把「spec §5 檢查點五問」列給使用者在 Electron 親驗，不自己判「對味」。
> ❌ 反例：自己認定「這個美術風格夠好了，收 Phase」→ 越權替使用者做了對味判斷。

## 4. 什麼訊號代表「方向錯了、該換路」而非重試

**換路（重想方法），若：**
- 同一個錯**換法試了兩輪還在原地**（熔斷點，見 `02` §4）。
- 為了讓某方案動，得一直加特例/繞路/改越來越多本來不相干的檔——方案在抵抗你。
- 修好 A 就冒出 B、修 B 冒出 C（打地鼠）＝抽象層選錯。
- 發現**計畫/spec 本身有錯**（自相矛盾、CSS bug、假設不成立）——這時要**連計畫一起修**，不是硬照錯計畫做。
- 驗證一直過不了，且每次失敗原因都不同。

**做法：** 停手 → 回到「要解的問題是什麼」重新拆 → 必要時升級模型或問使用者。不要在壞方案上繼續疊工。

> ✅ 正例：發現 plan 裡某 CSS clip-path 會裁掉 glow（plan 作者盲點）→ 連 plan 一起修，而非照著做出壞效果。
> ❌ 反例：測試一直紅，就一直微調 timeout/加 sleep/改斷言去遷就，沒問「是不是實作根本錯了」。系統性 debug 見 `superpowers:systematic-debugging`。

## 5. 品質底線怎麼驗（可勾選）

**交付前逐項驗，任一不過就不算過：**
- **程式碼**：`cd game && npm test` 全綠（記下數字）；動到 build 的跑 `npm run build`；動到體積的跑 `npm run check:size`（首載 gzip 上限 1465KB）。
- **測試有效**：測的是**行為**不是 mock；純邏輯有 TDD；邊角有覆蓋。
- **沒回歸**：測試數只增不減（除非明講刪了哪個且為什麼）。
- **視覺**：走 Electron CDP 截圖，**親眼看**過（不是假設），或明列「待使用者對味」+ 檢查點問題。
- **i18n**：新文案兩語系都補（key-align 守衛）；新 CJK 字會讓 tofu guard 紅 → `npm run fonts:build` 重生子集再 commit。
- **範圍紀律**：diff 只含這個任務該動的檔，沒夾帶無關重構/debug 殘留。

> ✅ 正例：hardening 收尾＝改 3 檔、313/313、build 乾淨、fresh-context agent 逐項推演 async 控制流 APPROVED。
> ❌ 反例：「測試我沒跑，但邏輯看起來對」——本專案 darkline.js 沒單測，唯一的把關就是跑 build + 獨立推演；跳過＝沒底線。

## 6. 補不了的怎麼辦（誠實條款）

harness 能用**拆解 + 獨立驗證 + 多樣本評審**補「執行品質」；但**模糊題與品味判斷補不了**。遇到時：

1. **先試補得動的部分**：把大判斷拆成可驗的小判斷（例：「好不好玩」拆成「操作有沒有卡頓／目標清不清楚／難度曲線」逐項查）。
2. **要第二意見**：多個獨立 agent 各給答案 → 評審選優/找矛盾（補得了「哪個較對」，補不了「使用者喜不喜歡」）。
3. **補不了就明說並上交**：直接告訴使用者「這是對味判斷，我做了 X 候選、控了 Y 風險，但對不對味要你判」——**不要假裝有客觀答案**。
4. **不確定就查；查不到就標註「不確定」**，不要編造 SHA/數字/事實。

> ✅ 正例：劇情候選產 A/B/C + 標各自風險，明說「推薦 B 摻 A，但選哪個要你拍板」。
> ❌ 反例：替使用者宣布「這個最好」並當定案往下做。
