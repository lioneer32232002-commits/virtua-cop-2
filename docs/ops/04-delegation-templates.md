# E — 派工 prompt 模板（填空即用）

> 給主對話模型直接複製套用。每份都含**派工三件套**（目標動機／驗收條件／回報格式）與建議 model+effort。
> 用法：複製對應區塊 → 填 `<…>` → 當 Agent tool 的 `prompt`。原則見 `02-model-dispatch.md`。
> 通用鐵則（每份都適用，不重寫在下面）：subagent 只回結論 + `檔案:行號`；長產物落檔回路徑；狀態用 DONE/DONE_WITH_CONCERNS/BLOCKED/NEEDS_CONTEXT；不確定就標註別編造。

---

## 1. 搜尋／探勘（agentType: `Explore`，model `haiku`~`sonnet`，effort `low`）

```
你的任務是找出並回報，不要改任何檔案。

## 目標與動機
我需要知道 <要找的東西>，因為 <為什麼要／接下來要拿它做什麼>。

## 搜尋範圍
- 從這裡找起：<路徑/關鍵字/慣例>
- 廣度：<medium＝幾處｜very thorough＝多處與多種命名>

## 驗收條件
- 回報 <要找的東西> 的位置與現況；找不到要明說「找不到」而非猜。
- 不要讀無關檔、不要把整檔貼回來。

## 回報格式
- 3–8 行結論；每個發現點附 `檔案:行號`。
- 若有多個候選，列出並標「最可能是哪個、為什麼」。
```

## 2. 實作（TDD）（model `sonnet`，effort `medium`；規格極完整且單檔可降 `haiku`/`low`）

```
你在實作 Task <N>：<名稱>。工作目錄 <path>，branch <branch>。用 Bash 跑測試：<指令>。

## 目標與動機
<這個功能是什麼、在整體哪個位置、為什麼要>。

## 任務全文（別叫我讀檔，貼在這）
<把 plan 對應 task 的完整步驟貼進來，含測試碼與實作碼>

## 慣例（一定遵守）
- 先 TDD：寫失敗測試 → 確認紅 → 實作 → 綠。
- 跟著既有風格（<指相關既有檔>）。
- 只動任務列的檔，不要順手重構別的。
- 影響玩家畫面的話：知道要走 Electron CDP 眼驗（見 electron/README.md），本 task 只要測試+build 綠。

## 驗收條件
- <列出的檔> 都建/改到位；`<npm test>` 全綠（回報數字）；沒回歸。
- 若新文案讓 tofu guard 紅 → `cd game && npm run fonts:build` 重生子集再 commit。
- 逐 task commit，訊息：<給定的 commit message>，結尾加 Co-Authored-By trailer。不要 push。

## 回報格式
狀態 + 實作了什麼 + 測試實際輸出（數字）+ 改了哪些檔 + commit SHA + 任何疑慮。
遇到和「任務全文」描述不符的既有碼 → 停下來回報，別硬猜。
```

## 3. 重構（行為零變更）（model `sonnet`，effort `medium`）

```
你在重構 <目標>，工作目錄 <path>。**行為必須零變更**——這是純結構調整。

## 目標與動機
<為什麼重構：去重／抽共用／可讀性>；重構後 <期望的結構>。

## 界線（很重要）
- 不改任何外部可觀察行為；現有測試必須**原封不動全綠**（不准改測試來遷就）。
- 範圍只在 <檔/區塊>，不要擴散。
- 若發現非重構不可的行為問題 → 停，回報，別自作主張改行為。

## 驗收條件
- `<npm test>` 全綠、數字和重構前一致；diff 只含結構調整。
- <若適用：跑 build / check:size 確認沒破包>。

## 回報格式
狀態 + 重構了什麼結構 + 測試數字（前後一致）+ 改了哪些檔 + commit SHA。
```

## 4. 研究（含虛構化/風控）（model `opus`（品味/風控）或 `sonnet`，effort `high`）

```
你在研究 <題目>，產出給使用者選/判對味的候選，不是最終定案。

## 目標與動機
<為什麼研究這個、要餵給哪個決策>。本專案內容分工：你研究+產草稿+控風險，使用者判對味。

## 約束（本專案特有）
- 史實改編一律**全面虛構化**（見 spec §13）；題材紅線：主題式影射 ✅／指名政黨領導人 ❌。
- 需要外部資料就查（WebSearch/WebFetch）；查不到就標「不確定」，不要編造事實/數字/引文。

## 驗收條件
- 產 <N> 個候選，每個標**各自風險**與取捨；給推薦但講明「最終要使用者拍板」。
- 長內容落檔到 <docs/…>，回路徑，別把整份塞回對話。

## 回報格式
結論 + 候選清單（各一句 + 風險）+ 推薦哪個為什麼 + 檔案路徑。哪些是查證的、哪些是推測，分清楚。
```

## 5. 審查（fresh-context 對抗式）（model `opus`，effort `high`~`max`）

```
你做獨立對抗式審查。**不要信建構者的回報，讀實際的碼與跑實際的驗證。** 只給意見，不改（除非明說順手修）。

## 審查對象
Commit/diff：<SHA 或範圍>。相關 spec/plan：<路徑或貼重點>。

## 要抓的（逐項讀碼確認，不是掃過）
- 偏離計畫：漏步驟、改了介面、命名不一致。
- 正確性 bug（尤其邊角、async 控制流、競態）。
- 測試缺口：純邏輯有沒有 TDD、邊角有沒有覆蓋、測的是行為還是 mock。
- 範圍：有沒有過度建置／夾帶無關改動。
- <若有檢查點判斷題：逐項給意見>。

## 驗證方式
- 檔案 read-back；程式碼**實際跑測試/build 看輸出**，不接受「應該會過」。
- 高風險處（刪大東西、控制流）用具體失敗情境推演。

## 回報格式
分三級：✅ 沒問題／⚠️ 建議改／🔴 必須改；每條給 `檔案:行號` + 具體理由 + 觸發情境。
最後一句結論：可否進下一步。
```

## 選模型速查（配合 `02-model-dispatch.md` 附錄艦隊）

| 任務型態 | 建議 model | effort |
|---|---|---|
| 搜尋/探勘/read-back | `haiku`（不夠再 `sonnet`） | `low` |
| 實作（規格完整） | `sonnet`（極簡單檔可 `haiku`） | `medium`（`low`） |
| 重構 | `sonnet` | `medium` |
| 研究/風控/品味草稿 | `opus`（或 `sonnet` 打底 + `opus` 把關） | `high` |
| 對抗式審查/最終把關 | `opus` | `high`~`max` |

> 現行「一律 Opus」政策下：全部用 `opus`，但**effort 仍照上表分級**，且照樣派工不自己下場。
