# F — 維護協議（未來模型怎麼安全更新這些檔）

> 這套 ops 檔要活著、跟著專案長。這份規定「誰能改什麼、踩坑寫哪、多長要精簡」。
> 核心原則：**改既有檔前先備份到 `docs/ops/_backup/<名>.<日期>.bak`**；新內容優先寫新檔；每次踩坑寫回對應檔。

---

## 1. 哪些可以自行改、哪些要先問使用者

| 檔 | 可自行改？ | 規則 |
|---|---|---|
| `01-harness-diagnosis.md` | ✅ 追加新坑 | 新坑可加；**刪既有條目要問使用者**（可能是硬教訓）。 |
| `02-model-dispatch.md` 附錄艦隊表 | ✅ 更新型號/ID | 型號更新照實改；**改「派工/驗證原則」本身要問使用者**。 |
| `03-judgment-rubrics.md` | ⚠️ 追加可、改判準要問 | 加新 rubric 可；**改既有判準的門檻/正反例要問使用者**（那是品味外化，動它等於動使用者的標準）。 |
| `04-delegation-templates.md` | ✅ 加新模板/微調 | 自由加常用模板；改既有模板留備份。 |
| `05-maintenance-protocol.md`（本檔） | ⚠️ 要問 | 改遊戲規則本身前先問。 |
| `06-letter-to-future-sessions.md` | ➕ 只追加 | 只在末尾追加「新一任的話」，不改前人寫的。 |
| `CLAUDE.md` | ⚠️ 高風險 | 每 session 必載。**改前一定備份 + 只做輕量路由/去重**，大改要問使用者。 |
| memory（`~/.claude/…/`） | ✅ 但守規則 | 見 §3。memory 是**本機**的，不跨機器，別把耐久紀錄只寫這。 |
| ROADMAP / HANDOFF / spec / plan | ✅ 照既有慣例 | 這些是專案權威記錄，照原本的更新方式。 |

**判準：** 要動的東西若是「使用者的標準/偏好/紅線/每 session 必載的檔」→ 先問。若是「事實更新、加新條目、加模板」→ 自己改，留備份。

## 2. 踩坑之後，教訓寫回哪、什麼格式

- **環境/工具坑**（build、CDP、OneDrive、資產）→ 寫回 `01-harness-diagnosis.md` 第三名清單 **和** memory `project_vc2_env_gotchas`。
- **判斷/流程教訓**（該升級沒升、該問沒問、驗證漏了）→ 寫回 `03-judgment-rubrics.md` 對應節，補一組正/反例。
- **使用者給的工作方式回饋** → memory `feedback_*`（一則一檔）+ 必要時 `03`。

**格式（每則坑）：**
```
- 症狀：<看到什麼現象>
- 原因：<根因，不是表象>
- 解法：<照這做就好，含指令/檔案:行號>
- 判準：<下次什麼訊號代表又踩到了，該直接套解法>
```
一句話能救未來一個 session 就值得寫。**寫「照這做」，不要寫「要小心」**（抽象提醒對弱模型沒用）。

---

## 3. memory 規則（呼應 `01` 第一名）

- **索引（MEMORY.md）每項一行**，≤120 字：一句話 + 指向 repo 權威檔。**不准**把變更史塞進索引。
- **一檔一則事實**；當前進度/已完成的事**不寫 memory**，寫 repo（ROADMAP 尾／最新 HANDOFF／commit）。
- memory 只留「從 code/git 看不出來的」：使用者偏好、工作方式回饋、踩過的環境坑、設計未定案的取捨。
- 寫前先查有沒有既有同主題檔，有就更新那則，別開重複。發現寫錯的記憶就刪。
- 累積過長（單則變 changelog）→ 用 `anthropic-skills:consolidate-memory` 或手動精簡，動前備份。

---

## 4. 累積多長要精簡

- **ops 檔單檔** > 約 300 行 → 檢查是否該把某節抽成子檔，主檔留路由。
- **HANDOFF**：只保留「最新一份」為接續點；舊的靠檔名日期自然沉底，不用刪，但接手只讀最新。
- **memory 單則** 超過幾行敘事 → 它已經變 changelog，精簡（見 §3）。
- **精簡原則**：留「還會被用到的規則與教訓」，砍「純歷史、已被 repo 記錄取代的細節」。精簡動作本身也要備份。

---

## 5. 每次收尾的最低動作

1. 這 session 學到的規則/坑，照 §2 寫回對應檔。
2. 改過的既有檔有沒有留備份？沒有就補。
3. 狀態寫進 repo（ROADMAP/HANDOFF）並 **push**——別只寫 memory（memory 不跨機器）。
4. 若 context 快用完：先落檔 + push 已完成的，未完成的寫進最新 HANDOFF 交接，再停。
