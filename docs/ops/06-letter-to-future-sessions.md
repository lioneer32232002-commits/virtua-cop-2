# G — 給未來 session 的信

> 寫於 2026-07-03，由 Opus 4.8 在收完 M3 Phase C 後寫。這封只追加、不覆蓋前人。
> 下面三件是你**沒問我、但我認為對這個環境最重要**的事，加上這套制度最可能怎麼爛掉、怎麼防。

---

## 三件最重要的事

### 1. repo 是唯一的長期記憶；memory/ 是本機的，換機器就歸零

這個環境**跨多台機器**（用戶各台各自 `git clone`）。Claude Code 的 memory 存本機 `~/.claude/…`，**不跟著 repo 走**。所以：
- 任何「下次還要用到」的狀態、決定、教訓——**寫進 repo 並 push**（ROADMAP／HANDOFF／spec／這套 ops 檔），不要只寫 memory 或只留在對話裡。
- 這不是建議，是**存活條件**：本專案很多混亂都源自「寫在記憶/對話沒進 repo → 換機器 → 全沒了」。
- 收尾沒 push＝這 session 對下一台機器等於沒發生。

### 2. 這個專案的真瓶頸是「品味/craft」，那正是模型補不了的——不要自動幫使用者決定對味

程式、測試、重構、體積守衛這些**已經很順**（有 TDD、有守衛、有 Electron CDP 驗證）。剩下決定「這作品好不好」的，是**美術對味、玩起來爽不爽、劇情/文案對不對味**——這些**只有使用者能判**，分工白紙黑字。
- 最大的未完短板是 **sprite/美術內容升質**（Milestone E：現在敵人是單張 billboard、主角無形象）。那是**內容協作**任務（Claude 產候選、使用者拍板），不是寫 code 就能收的。
- 千萬別為了省一次 round-trip，自己判「這風格夠好了、收」。那是把使用者唯一不可替代的價值搶走，而且你判不準。真心想推進，就**產好候選 + 控好風險 + 把選擇權留給使用者**。

### 3. OneDrive + git 很脆，把工作副本當半敵意環境

工作目錄在 OneDrive 底下。**鎖檔、半同步**會弄壞 git（有前科）。
- **不要用 OneDrive 把工作副本同步到另一台機器**——每台各自 clone。
- worktree 刪不動就 `rm -rf`；遇到詭異的 git 狀態先想「是不是 OneDrive 在半同步」。
- 大量檔案操作前後留意鎖檔；不確定就先 `git status` 確認乾淨。

## 這套制度最可能怎麼爛掉，怎麼防

**最可能的退化：ops 檔自己變成「root HANDOFF.md 第二」——沒人更新、和現實脫節，然後某個未來模型讀了過期規則、照做、出錯。**（這正是 `01-harness-diagnosis.md` 第二名的坑，遞迴套到制度本身。）

具體三條退化路徑 + 防法：

1. **路由斷掉 → 整套變暗檔。** 若哪天 CLAUDE.md 那行指向 `docs/ops/` 的路由被刪/改壞，未來 session 根本不會載入這套，等於不存在。
   **防：** CLAUDE.md 的 ops 路由是**命根**，動 CLAUDE.md 時務必保留它（見 `05` §1 把 CLAUDE.md 列高風險）。

2. **規則和現實脫節 → 誤導。** 型號改了、政策從「一律 Opus」變了、坑的解法變了，但 ops 檔沒跟上。
   **防：** ops 檔盡量寫**原則**（爛得慢）而非**當前狀態**（爛得快）；會變的東西（型號 ID、政策）集中在有日期標記的小節，改起來一眼看到。每到里程碑收尾，派一個 fresh-context agent 拿 ops 檔對一次現實。

3. **沒人跑維護動作 → 慢慢腐。** `05` §5 的收尾動作被跳過，教訓沒寫回，memory 又長回 changelog。
   **防：** 把「收尾照 `05` §5 落檔+push」當硬步驟，不是選配。踩到坑當下就寫回（趁熱），別想著「之後再整理」——之後不會來。

---

## 一句話總結給你

**把狀態推進 repo、把對味留給使用者、把踩到的坑趁熱寫回。** 這三件做到，你就算比我弱，也不會讓這個專案退步。
